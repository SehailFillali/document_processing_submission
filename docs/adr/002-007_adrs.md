# ADR 002: Storage Strategy

## Status
Accepted

## Context
We need to choose storage for both document blobs and extracted data.

## Decision
MVP uses SQLite + Local File System. Production uses GCS + BigQuery.

## Options Considered

| Option | Pros | Cons | Best For |
|--------|------|------|----------|
| **SQLite + Local (MVP)** | Simple, no setup | Not scalable, local only | Development |
| PostgreSQL | Production ready | Requires infrastructure | Production MVP |
| **GCS + BigQuery (Production)** | Scalable, managed | Cost, complexity | Production |
| DynamoDB | Fast, serverless | Expensive for analytics | Real-time apps |

## Consequences
- Local storage for MVP enables quick iteration
- GCS/BigQuery ready when scale needed

# ADR 003: LLM Choice

## Status
Accepted

## Context
We need to select an LLM provider for document extraction.

## Decision
Use PydanticAI with Gemini via API key as primary, with OpenAI/VertexAI as alternatives via adapter swap.

## Options Considered

| Option | Pros | Cons | Best For |
|--------|------|------|----------|
| **PydanticAI + Gemini (Chosen)** | Enforced schemas, easy API key | Context limits | Most use cases |
| PydanticAI + VertexAI | Enterprise ready | Requires GCP setup | Enterprise |
| OpenAI + Instructor | Strong schemas | Cost | Complex reasoning |
| Claude + SDK | Excellent reasoning | Different API style | Complex docs |

## Consequences
- PydanticAI enforces output schemas automatically
- Easy to swap to VertexAI for enterprise deployment

# ADR 004: State Machine

## Status
Accepted

## Context
Document processing has multiple stages that need state tracking.

## Decision
Use Pydantic Graph for state machine with 3 nodes: Preprocess → Extract → Validate.

## Options Considered

| Option | Pros | Cons | Best For |
|--------|------|------|----------|
| **Pydantic Graph (Chosen)** | Type-safe, native | Newer library | Complex workflows |
| Airflow | Mature, powerful | Heavy, complex | Batch processing |
| Temporal | Durable, reliable | Complex setup | Mission critical |
| Simple Functions | No overhead | No state tracking | Simple pipelines |

## Consequences
- Type-safe state transitions
- Easy to add human-in-the-loop for low confidence

# ADR 005: Event-Driven Design

## Status
Accepted

## Context
We need async processing to not block API responses.

## Decision
Use async message queue with Pub/Sub pattern. API publishes event, worker consumes.

## Options Considered

| Option | Pros | Cons | Best For |
|--------|------|------|----------|
| **Async Memory (Dev)** | Simple | Not persistent | Development |
| **Google Pub/Sub (Prod)** | Scalable, managed | Requires GCP | Production |
| Kafka | High throughput | Complex ops | High volume |
| RabbitMQ | Flexible | Not managed | Hybrid |

## Consequences
- Non-blocking API responses
- Scalable worker processing

# ADR 006: Horizontal Scaling

## Status
Accepted

## Context
System needs to handle 10x and 100x volume growth.

## Decision
Cloud Run with auto-scaling for API, Pub/Sub for async processing.

## Scaling Strategy

| Scale Level | Architecture | Max Documents/Day |
|-------------|--------------|-------------------|
| **10x** | Cloud Run + Pub/Sub | ~10,000 |
| **100x** | Cloud Run + GCS + BigQuery | ~100,000 |

## Horizontal Scaling Approach
- Cloud Run: 1-100 instances, auto-scale on requests
- Pub/Sub: Unlimited throughput with backlog
- BigQuery: Petabyte-scale analytics

# ADR 007: Error Handling

## Status
Accepted

## Context
LLM extraction can fail partially or completely.

## Decision
Structured error response with partial success support. Dead Letter Queue for failed messages.

## Error Categories

| Category | Handling | User Impact |
|----------|----------|-------------|
| Validation | Fail fast, reject | Immediate feedback |
| LLM Timeout | Retry with backoff | Delayed response |
| Partial Extract | Return valid fields + errors | Partial data |
| Complete Failure | DLQ + notification | Manual review |

## Validation Strategy
- Pydantic strict mode for input validation
- Confidence scores from LLM
- Explicit MissingField for absent data
